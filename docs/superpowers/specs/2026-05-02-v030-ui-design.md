# App Launcher v0.3.0 — UI Design Spec

**Date:** 2026-05-02  
**Status:** Approved  
**Scope:** Frontend UI for three v0.3.0 backend features — all Rust commands already implemented and uncommitted in the working tree.

---

## Overview

Three UI surfaces to build:

1. **Config window tabs** — splits the existing group editor into Group / Settings tabs
2. **Settings tab** — hotkey configuration + import/export config
3. **Per-item targeting** — expandable item rows with a screen-coordinate location picker

No new Rust commands are needed beyond minor changes to `config.rs` (swap `launch_position`/`launch_monitor` for `launch_x`/`launch_y`) and `launcher.rs` (simplify positioning to use raw coords). The picker overlay requires one new Rust command (`start_location_picker`) and a new `picker.html` page.

---

## 1. Config Window Tabs

### HTML structure (`src/config.html`)

Add a tab bar immediately inside `.config-window`, before all existing content:

```html
<div class="tab-bar">
  <button class="tab active" data-tab="group">Group</button>
  <button class="tab" data-tab="settings">Settings</button>
</div>
<div id="tab-group">
  <!-- all existing config-window content -->
</div>
<div id="tab-settings" style="display:none">
  <!-- new settings panel -->
</div>
```

### Tab switching (`src/config.js`)

- On `DOMContentLoaded`, read `new URLSearchParams(window.location.search).get('tab')`. If `'settings'`, activate the Settings tab.
- Tab click handler: toggle `.active` on buttons, toggle `display` on panels, call `fitWindow()`.

### Opening Settings from the widget

Add "⚙️ App Settings…" to the native right-click context menu in `lib.rs` (after the color items, before "Close"). Its handler calls `open_config_window` with `"config.html?tab=settings"` — same pattern used to open the group editor.

## 2. Settings Tab

### HTML (`src/config.html`, inside `#tab-settings`)

```
┌─────────────────────────────────────────┐
│  GLOBAL HOTKEY                          │
│  [ Ctrl+Alt+Space          ] [ Save ]   │
│                                         │
│  CONFIG FILE                            │
│  [ ⬆ Export Config ]  [ ⬇ Import Config]│
└─────────────────────────────────────────┘
```

- Hotkey input: text field, placeholder `Ctrl+Alt+Space`, populated from `config.hotkey` on load
- Save button: calls `invoke('set_hotkey', { hotkey })`, shows brief "Saved ✓" confirmation inline
- Export / Import buttons: call existing `invoke('export_config')` / `invoke('import_config')`
- `fitWindow()` called after panel becomes visible

### JS (`src/config.js`)

`initSettingsTab()` called during `init()`:
- Loads current config via `invoke('get_config')`, sets hotkey input value
- Wires Save, Export, Import button handlers

## 3. Per-Item Targeting

### Config struct changes (`src-tauri/src/config.rs`)

Remove `launch_position: Option<String>` and `launch_monitor: Option<u32>` (both are only in uncommitted code — no migration needed). Add:

```rust
#[serde(default)]
pub launch_x: Option<i32>,
#[serde(default)]
pub launch_y: Option<i32>,
```

### Launcher changes (`src-tauri/src/launcher.rs`)

`position_window_for_item` simplified: if `launch_x` and `launch_y` are both `Some`, call `SetWindowPos` with those absolute coordinates directly. Remove all named-position math (`"top-left"`, `"center"`, etc.) and monitor-index logic.

### Item row UI (`src/config.js` — `renderItems`)

Each item row gets a `›` chevron before the `✕`. Clicking the chevron toggles an expand panel below the row:

```
Launch at  [ x: 1240, y: 320 · Display 2 ]  [✕]  [📍 Pick]
```

- **Coord display**: green monospace text. Shows `x: N, y: N · Display N` when set; `not set` (muted) when empty.
- **✕ button**: clears `launch_x`/`launch_y` on the item object, re-renders row.
- **📍 Pick button**: calls `invoke('start_location_picker')`, then listens for `location-picked` event. On receipt, sets coords on item, re-renders, calls `fitWindow()`.

`currentItems` entries gain `launch_x` and `launch_y` fields (may be `null`). These are included when building the `group` object passed to `save_group`.

### Location picker overlay

#### New files: `src/picker.html` + `src/picker.js`

`picker.html`: minimal page — transparent background (`background: transparent`), `cursor: crosshair`, a fixed tooltip div near the cursor, a fixed hint bar at top ("Click anywhere to set launch position · Esc to cancel").

`picker.js`:
- `mousemove`: update tooltip text with `e.screenX, e.screenY`. Determine display name by calling `invoke('get_monitors')` once on load and finding which monitor rect contains the cursor.
- `click`: `invoke('finish_location_picker', { x: e.screenX, y: e.screenY })`. Do not emit directly — let Rust re-show the config window first, then emit the event, to avoid a race where the config window receives the event before it's visible.
- `keydown Escape`: `invoke('cancel_location_picker')`.

#### New Rust command: `start_location_picker` (`src-tauri/src/lib.rs`)

1. Hide the config window (`app.get_webview_window("config").hide()`).
2. Read virtual screen bounds via `GetSystemMetrics(SM_XVIRTUALSCREEN / SM_YVIRTUALSCREEN / SM_CXVIRTUALSCREEN / SM_CYVIRTUALSCREEN)`.
3. Create a new `WebviewWindow` at the virtual screen origin with full virtual screen dimensions, transparent background, always-on-top, `skip_taskbar: true`, `decorations: false`, loading `picker.html`.

#### New Rust command: `finish_location_picker { x: i32, y: i32 }`

1. Close the picker window.
2. Re-show the config window (by label `"config"`).
3. Emit `location-picked` event targeted at the `"config"` window label with payload `{ x, y }`. Targeting by label (not broadcast) avoids the event firing in any other open window.

#### New Rust command: `cancel_location_picker`

1. Close the picker window.
2. Re-show the config window. No event emitted.

#### Capabilities (`src-tauri/capabilities/default.json`)

Add permissions required for creating/closing windows and emitting events from the picker.

## 6. Out of Scope

- **Virtual desktop targeting** (`launch_desktop`) — deferred to v0.4.0. Config field stays but has no UI or implementation.
- **Hotkey recording** (click a key combination to set) — user types the hotkey string manually for now.
- **Multi-monitor picker on Linux/macOS** — picker uses `GetSystemMetrics` (Windows only). Non-Windows: picker window covers primary monitor only.
